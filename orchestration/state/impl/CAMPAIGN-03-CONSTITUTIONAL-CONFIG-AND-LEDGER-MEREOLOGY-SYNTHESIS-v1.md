Campaign 03 found that the shell is not blocked by missing macro doctrine.

1. one constitutional config source-of-truth architecture

## 2. Main Adjudications

### 2.1 Custom instructions are witnesses, not the final source file

The imported chat-app custom-instruction family remains the highest-resolution behavioral witness currently available.

It should not become the direct render source for lower-level config.

The lawful chain is:

`canon -> Sigma -> constitutional config kernel -> derivative config surfaces -> rendered vendor artifacts -> config ledgers and reports`

This preserves behavioral fidelity without letting vendor phrasing drift become shell law.

### 2.2 The shell needs typed behavior atoms and surface profiles

The stable cross-surface essence captured by the custom-instruction family should be decomposed into:

- behavior atoms
- anti-pattern suppressions
- surface profiles
- projection rules
- renderability constraints

That atom layer should become the direct basis for future CLI constitutions, harness veneers, skills, hooks, schemas, and templates.

### 2.3 Mereology is already present, but still implicit

Campaign 03 confirmed that the validated repo already realizes six strata unevenly:

1. atoms
2. configural modules
3. instructional structures
4. agentic entities
5. platform and workflow architectures
6. ecosystem surfaces

The repo is strongest at instructional structures and workflow architectures.
It is weakest where atoms, modules, and entities are still carried implicitly inside broader prose or code.

### 2.4 Office identity is durable; harness binding is metadata

Office directories and office identity remain constitutional.
Harness binding is mutable, typed, repo-ratified metadata.

The lawful chain is:

`AGENTS.md -> office-harness binding law -> per-office metadata -> rendered effective registry -> validator reports -> downstream projections`

This prevents directory churn or veneer drift from silently redefining offices.

### 2.5 Live-ledger widening needs one family contract first

The tributary control plane remains the proof artifact.
It should not be copied blindly into many repo-native ledgers.

The generalized family rule is:

`repo-ratified law -> exocortex append-only substrate -> derived current-state surface -> repo-ratified synthesis snapshot -> optional dashboard and ontology projections`

This keeps repo sovereignty intact while allowing higher-churn operational state to live where it belongs.

### 2.6 Ratification pointers should roll out schema-law first

Campaign 03 converged on a non-breaking transition path:

1. ratify schema-law transition
2. add validator compatibility
3. publish header and row-shape revision last

That preserves the live tributary proof while explicitly separating `authority_bound` from `informative_only`.
