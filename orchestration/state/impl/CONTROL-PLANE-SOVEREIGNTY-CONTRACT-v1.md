# Control-Plane Sovereignty Contract v1

**Status**: live-v1
**Class**: implementation law
**Purpose**: define the specialization boundary between repo, exocortex, and ontology so scale does not create hidden sovereigns

## Compact Contract

- repo ratifies
- exocortex coordinates
- ontology projects

Nothing direction-bearing becomes shell truth until repo ratification exists.

## Layer Burdens

| Layer | Allowed burden | Forbidden drift |
|---|---|---|
| Repo | law, doctrine, office burden, program direction, authority-bearing registries, schema law, and any change that alters what downstream surfaces may rely on | delegating ratification to tools, dashboards, registries, or graph shape |
| Exocortex | live control-plane state, routing, receipts, verification, bounded automation, transient operational context | becoming the only home of policy, legitimacy, teleology, or permission |
| Ontology | typed projections, lineage, relations, statuses, conflict visibility, query surfaces over repo and exocortex state | turning entity classes, relation names, or field families into hidden doctrine |

## Ratification-Pointer Rule

Any authority-bearing registry row or schema change must carry a ratification pointer to the repo artifact that authorizes it.

Authority-bearing means the artifact can change or imply:

- legitimacy
- teleology
- role burden
- disposition
- promotion state
- allowed reliance
- semantic meaning for downstream operators or systems

Minimum pointer fields:

- `ratification_pointer`
- `ratified_by_artifact_path`
- `ratified_by_artifact_id`
- `ratified_at`

Rules:

1. If a registry row or schema change lacks a ratification pointer, it is informative only and must not be treated as law.
2. If a control-plane artifact changes execution only, it may exist without ratification, but it must not redefine meaning.
3. If a schema addition, rename, merge, split, or field-family change could alter interpretation, it requires repo ratification before downstream adoption.
4. Ontology projections may repeat the pointer; they may not replace it.
5. In conflicts, repo ratification beats control-plane freshness and schema convenience.

## Live Promotion Path

`event or proposal -> receipt or draft registry/schema change -> repo ratification artifact -> pointer attached -> exocortex/ontology projection`

## Current Effect

This contract is immediately binding on:

- control-plane registry law
- future live-ledger family design
- authority-bearing schema evolution
- any attempt to let dashboards or projections silently redefine shell meaning
