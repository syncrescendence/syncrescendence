## Pattern

A good orientation surface should answer, in one place:

- what the system is
- what the current constellation or office map is
- what the main execution surfaces are
- what the state machine or work lifecycle looks like
- what the source of truth is

## Why It Survives

The predecessor cockpit was overloaded, but it got one thing very right: a new entrant needed a fast 30,000-foot briefing that connected machine layout, roles, routing, and repository sovereignty.

## Successor-Shell Translation

The successor shell should preserve the orientation function without restoring obsolete details such as stale account matrices, dead lanes, or deprecated topologies.

Good orientation surfaces should be:

- concise
- current
- explicit about authority
- separate from detailed playbooks and runtime state
