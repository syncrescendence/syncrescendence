## 1. Decision

The shell should not derive lower-level configuration surfaces by copying, paraphrasing, or diffing vendor custom-instruction text directly.

It should use this authority path:

`canon -> Sigma -> constitutional config kernel -> derivative config surfaces -> rendered vendor artifacts -> config ledgers and reports`

This resolves the central tension:

- custom instructions remain the highest-resolution behavioral witness
- they do not become the direct render source for hooks, skills, schemas, or other lower surfaces
- Sigma distills their stable cross-surface behavior into shell-universal atoms
- a typed config kernel governs how those atoms project into each lower surface

The critical rule is:

**custom instructions are the highest-resolution witness, not the lowest-level source file**

If the shell renders directly from provider text, it will duplicate doctrine and import vendor phrasing drift into every lower surface.
If it renders from a constitutional config kernel, each lower surface can stay native to its constraint surface without inventing a second personality.

## 2. Canonical Source Layers

Only the first three layers below may introduce durable behavioral doctrine.
Everything after that is subordinate, local, projected, or append-only.

### 2.1 Layer A: canon

Canon holds the macro law that must survive every vendor and harness shift:

- strategic telos
- sovereignty boundaries
- office boundaries and burdens
- ratification law
- compaction law
- trust-boundary law
- shell-universal anti-drift rules

Canon does not hold provider syntax, field-packing tactics, or harness-specific operational recipes.

### 2.2 Layer B: Sigma

Sigma holds the translation doctrine that turns macro law into reusable secondary config law:

- cross-surface behavioral invariants
- custom-instruction interpretation doctrine
- config-source architecture doctrine
- projection and render law
- anti-pattern doctrine
- ledger family doctrine where config state must become append-only

The exact imported chat custom-instruction texts remain witness substrates under Sigma references.
Sigma interprets them; it does not let them quietly become sovereign.

### 2.3 Layer C: constitutional config kernel

This is the missing layer that should become the direct source for lower-level config generation.
It should be repo-native, typed, and ratified.

It should contain:

- behavior atoms
- anti-pattern suppressions
- surface profiles
- projection priorities
- source bindings
- renderability constraints
- explicit omission rules for surfaces that cannot express every atom

This layer is where the shell says:

- which enduring behaviors are universal
- which surfaces must carry which subset
- what compression is lawful
- what packaging is vendor-specific
- which artifacts are hand-authored versus projected

## 3. Derivative Config Layers

Everything below the config kernel is derivative.
Some derivative surfaces remain hand-authored.
Some are rendered.
None may create new macro personality.

### 3.1 Playbooks

Playbooks are locally canonical but globally derivative.

They should hold:

- office or surface workflow doctrine
- burden-specific tactics
- capability boundaries
- escalation logic
- repeated local anti-patterns
- surface-native examples

They should not restate shell identity, macro office law, or cross-surface behavior law as if they originated there.

### 3.2 Skills

Skills are on-demand instructional modules.

They should hold:

- bounded task protocols
- checklists
- domain-specific heuristics
- progressive-disclosure instructions
- narrow operational constraints

They should not define shell personality, federal office law, or hidden policy.
The body may be hand-authored, but its identity and doctrine must trace back to playbooks or the config kernel.

### 3.3 Hooks

Hooks are executable enforcement and audit surfaces.

They should hold:

- deterministic policy checks
- gating logic
- notifications
- audit emission
- transformation or sanitation steps when explicitly lawful

Hooks should not carry prose personality or duplicate doctrine in long natural-language blocks.
They should enforce policy, not invent it.

### 3.4 Schemas

Schemas are typed contracts for structured config and ledgers.

They should hold:

- field definitions
- enums
- required pointers
- validation rules
- versioning logic
- structured omission or capability flags

Schemas should not carry personality, motivational text, or vendor lore beyond the minimum needed to define a field.

### 3.5 Templates

Templates are reusable artifact skeletons.

They should hold:

- placeholders
- section order
- required headings
- minimum metadata
- expected return shape

Templates should not become stealth doctrine stores.
They scaffold lawful repetition; they do not author new law.

### 3.6 Ledgers

Ledgers are append-only or mutation-traceable config state organs.

They should hold:

- render events
- validation findings
- drift detections
- promotion receipts
- freshness markers
- ratification pointers for authority-bearing state changes

Ledgers should never become the place where the shell's behavior is redefined.
They preserve state and history, not doctrine.

## 5. Constitutional Config Kernel Contents

The config kernel should be the first surface that lower-level config renderers read directly.

Its minimum contents should be:

### 5.1 Behavior atoms

At minimum, the first atom family should include:

- response scaling by task complexity
- observation, inference, and speculation separation
- active, direct prose
- explicit anti-pattern suppression
- honest pushback over servile agreement
- cognitive accommodation by structure rather than ornament
- provenance and receipt discipline
- compaction pressure toward reusable doctrine

These come from the CI witness family plus the shell's broader constitutional law.

### 5.2 Surface profiles

Each managed surface should declare:

- surface id
- surface family
- scope
- owner office or doctrine family
- max packing or verbosity constraints
- whether the surface is hand-authored, rendered, or append-only
- whether it supports free prose, structured fields, executable policy, or typed state
- what atom families are mandatory, optional, or inexpressible there

### 5.3 Projection priorities

The kernel should specify which elements compress first when a surface is constrained.

For example:

- preserve behavioral atoms before explanatory prose
- preserve anti-pattern suppressions before examples
- preserve explicit role boundaries before stylistic flourishes
- preserve provenance and uncertainty rules before convenience wording

### 5.4 Explicit omission protocol

If a surface cannot carry a universal atom, the omission must be explicit in the registry.
Silent loss is drift.

Examples:

- a tiny vendor field may require atom compression
- a hook file may not express prose atoms at all
- a schema may represent an atom only as a field or enum rather than natural language

## 6. Render And Projection Rules

### 6.1 General rules

1. No projection is edited as the primary source if a lawful upstream source exists.
2. No lower surface may introduce new macro personality.
3. Every managed projection must declare its authoritative source artifacts.
4. Every managed projection must declare its renderer and validator.
5. Every authority-bearing config registry or schema change must carry a ratification pointer.
6. Compression is allowed; silent doctrinal mutation is not.
7. Vendor-specific wording may change; atom coverage may not, unless an omission is explicitly declared.
8. Lower surfaces may add capability-binding facts, not new constitutional law.

### 6.2 Playbook projection rule

Playbooks may elaborate workflow and burden but must inherit universal behavioral atoms from the kernel.
They may tune surface grain.
They may not redefine the shell's core identity.

### 6.3 Skill projection rule

Skills should derive from playbooks, repeated communications patterns, or validated procedural wins.
They should carry task-local instruction only.
If a skill starts carrying general behavioral law, it should compact upward into Sigma or the config kernel.

### 6.4 Hook projection rule

Hooks should be rendered from policy specs and schemas where possible.
Their natural-language explanations should stay minimal.
A hook is a trust-boundary instrument, not a personality surface.

### 6.5 Schema projection rule

Shell-owned schemas are canonical for structured state.
Vendor-owned schemas remain external constraints.
The shell may mirror vendor schema facts for validation, but those mirrors are derivative references, not canon.

### 6.6 Template projection rule

Templates may pre-populate lawful headings and required metadata.
They should not freeze incidental wording that later becomes hard to update.
Where possible, templates should inherit section contracts from the config kernel or schema surface.

### 6.7 Ledger rule

Config ledgers should record:

- when a projection rendered
- which source artifacts it used
- what validator version checked it
- whether drift or manual override was detected
- which ratified artifact authorized the current state

They should never be treated as the source that future renders read as doctrine.

## 7. Shell-Universal Versus Vendor-Specific

### 7.1 Shell-universal

These belong in canon, Sigma, or the config kernel:

- behavioral atoms
- anti-pattern suppressions
- office and burden boundaries
- provenance and uncertainty rules
- compaction and promotion rules
- ratification and trust-boundary rules
- shell-owned schemas and ledger contracts
- surface classification and edit-policy rules

### 7.2 Vendor-surface specific

These belong in playbooks, manifests, renderers, or projected artifacts:

- field names and field splits
- XML, Markdown, YAML, or JSON syntax details
- provider token or character budgets
- local adapter paths
- hook registration APIs
- permission DSLs
- skill packaging locations
- model names, provider toggles, and runtime-specific settings
- provider quirks such as:
  - ChatGPT two-field packing
  - Claude large preferences block
  - Grok global versus named-agent partition
  - Gemini web-app versus AI Studio split

The vendor layer may shape packaging, not doctrine.

## 10. Net Verdict

The correct source-of-truth architecture is not:

- canon plus a pile of vendor texts
- or one monolithic universal instruction slab
- or freehand local config per surface

It is:

- canon for macro law
- Sigma for translation doctrine
- a constitutional config kernel for behavior atoms and surface profiles
- derivative playbooks, skills, hooks, schemas, templates, and ledgers governed by that kernel
- rendered vendor artifacts treated as replaceable projections

That architecture keeps the CI family in its correct place:

- highest-resolution behavioral witness
- not hidden sovereign
- not direct render source
- not duplicated doctrine
