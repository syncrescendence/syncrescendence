## Returned Content

The governing contract is:

- repo ratifies
- exocortex coordinates
- ontology projects

Non-duplication exists to stop convenience layers from becoming hidden sovereigns. The repo remains the only ratification surface. The exocortex may carry live operational state. The ontology may normalize and expose typed views of ratified and operational state. Neither exocortex nor ontology may silently acquire authority over doctrine, legitimacy, priority, or executive direction.

## Three-Column Contract

| Repo authority | Exocortex coordination state | Ontology projection |
|---|---|---|
| Constitutional law, doctrine, durable shell rules, office burdens, program decisions, executive steering, migration law, and any artifact that changes what the shell is allowed to do or believe | Live connector topology, routing state, execution receipts, bounded automations, verification progress, surface-local operating context, and transient control-plane state needed to run the external shell | Typed entities, relations, registries, statuses, and lineage views derived from repo artifacts and exocortex snapshots for query, audit, reconciliation, and graph traversal |
| Ratified manifests, contracts, playbooks, validated patterns, promotion decisions, and disposition records | Connector manifests and control-plane status as operational snapshots, provided they remain subordinate to repo law and exclude secrets | Normalized projections of repo and exocortex artifacts such as `ExocortexRegistry`, `ExocortexTeleology`, `ExocortexConnectorManifest`, and `ExocortexControlPlaneStatus` |
| The sole place where direction-changing outputs become institutional truth | The place where work happens across Slack, Notion, Linear, ClickUp, Cloudflare, model brokers, and other external surfaces | The place where projected entities become legible and queryable, but never self-ratifying |

## Non-Duplication Rules

### Repo

1. If an artifact changes doctrine, legitimacy, priority, disposition, promotion class, office burden, or surface policy, it must exist in repo form.
2. Repo may reference exocortex or ontology state, but it may not delegate ratification to either.
3. Repo records the lawful outcome, not every transient control-plane fluctuation.
4. Repo is the only layer allowed to carry silent-default authority. If a conflict exists, repo wins until a new repo artifact ratifies change.

### Exocortex

1. Exocortex may store only operationally necessary state: topology, routing, verification, receipts, execution traces, and bounded automation context.
2. Exocortex must not be the only home of any decision that changes policy or direction.
3. Exocortex state may be current; it is not therefore authoritative.
4. Exocortex must not silently promote mutable GUI state, chat state, or connector affordances into law.
5. Exocortex-native records require receipt or snapshot forms when they materially affect repo-governed surfaces.

### Ontology

1. Ontology may project structure, classification, linkage, and status; it may not originate doctrine.
2. Ontology entity classes, relation types, and field families must not become de facto law by schema convenience.
3. Every projected claim that matters must resolve back to a repo ratification artifact or an explicitly subordinate exocortex snapshot.
4. Ontology may compress and normalize names, but it may not reinterpret meaning without a repo-ratified contract amendment.
5. Ontology is a query surface only. It must not become a hidden write path for institutional truth.

## Promotion Rules For Direction-Changing Outputs

1. Any output that changes shell direction, legitimacy, priority, role burden, migration disposition, teleology, or surface policy is provisional until it lands in a repo artifact.
2. Exocortex-originated changes must follow this sequence:
   `surface event or operator action -> typed receipt or status snapshot -> repo ratification artifact -> ontology re-projection`
3. Ontology-originated synthesis, anomaly detection, or graph inference may propose changes, but proposals remain advisory until ratified in repo.
4. Control-plane changes that alter what a surface is for, who may rely on it, or whether it is an identity root require a repo update before downstream consumers may treat the change as lawful.
5. Promotion must be explicit, not inferred. A query result, dashboard state, or verified connector badge does not count as ratification.
6. If time pressure requires acting before ratification, the exocortex may execute a bounded emergency action, but the action remains temporary until repo receipt and ratification are recorded.

## Required Lineage Links

Every projected ontology entity that represents shell-relevant truth must carry lineage sufficient to walk back to authority.

Required lineage fields:

- `source_artifact_path`: repo or snapshot artifact path
- `source_artifact_id`: stable artifact identifier or snapshot identifier
- `source_artifact_class`: one of `repo_ratified`, `exocortex_snapshot`, or `derived_projection`
- `ratification_pointer`: required for any direction-bearing or doctrine-bearing entity
- `projection_timestamp`: when ontology emitted the projection
- `upstream_snapshot_timestamp`: when exocortex or repo state was captured

Lineage law:

1. If `source_artifact_class = repo_ratified`, the entity may inherit authority only within the meaning defined by that artifact.
2. If `source_artifact_class = exocortex_snapshot`, the entity is operationally informative but non-ratifying.
3. If `source_artifact_class = derived_projection`, it must point to the upstream repo or exocortex records it summarizes and must not be treated as a new authority object.
4. No projected entity representing teleology, ownership, legitimacy, migration disposition, or surface role may exist without a repo ratification pointer.
5. Multiple-source entities must preserve dual witness rather than collapse conflicts silently.

## Top Failure Modes If Control-Plane Convenience Outruns Authority Law

1. Verified-state usurpation. Teams begin treating connector verification or topology completeness as permission to rely on a surface beyond what repo law authorizes.
2. Schema doctrine drift. Ontology classes and relation names harden into institutional meaning without any corresponding repo contract.
3. GUI-sovereignty creep. Slack channels, Notion docs, Linear workflows, or dashboard toggles become the place where real decisions happen, with repo updated later or not at all.
4. Silent teleology mutation. A surface's purpose, identity role, or dependency status changes in exocortex registries and downstream users absorb it as truth without repo ratification.
5. Lineage amnesia. Projected entities lose their pointer to ratified artifacts, so query users can no longer distinguish law from convenience.
6. Emergency-state ossification. Temporary operational exceptions persist in control-plane state long enough that they become customary policy without an explicit promotion ceremony.
7. Conflict masking. Ontology normalization flattens disagreement between repo doctrine and exocortex observations, leaving operators with a false singular truth.

## Bottom Line

The contract is not a redesign of the exocortex. It is a boundary law:

- repo holds ratified meaning
- exocortex holds live coordination
- ontology holds typed projection

Nothing may move silently uphill. Operational facts may project outward and upward for visibility, but direction-changing meaning must always return to repo for ratification before the shell treats it as truth.
