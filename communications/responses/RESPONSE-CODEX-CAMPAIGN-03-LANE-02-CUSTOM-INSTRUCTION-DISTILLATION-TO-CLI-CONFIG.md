# Response — Codex Campaign 03 Lane 02 Custom-Instruction Distillation To CLI Config

## Executive Answer

The chat-app CI family should be treated as a high-resolution witness corpus, not as the literal text to paste into CLI surfaces.

Its durable payload is the behavioral kernel:

- response scale follows task complexity
- observation, inference, and speculation stay separated
- prose stays direct and affirmative
- anti-patterns are named and suppressed
- the model collaborates through honest pushback
- operator cognition is accommodated structurally, not cosmetically

That kernel should be distilled downward through a repo-native projection stack:

`chat-app CI witnesses -> constitutional atoms -> harness projections -> executable surfaces -> validators and render ledger`

The further down the stack a surface sits, the less philosophy it should carry and the more it should become explicit structure, executable policy, or output schema.

## Distillation Law

### 1. Witness, not sovereign source

Imported chat custom instructions are the clearest available behavioral captures, but they are still rendered veneers for vendor UI constraint surfaces.

They must therefore be treated as:

- evidence of desired behavior
- input to distillation
- parity-check witnesses

They must not be treated as:

- the top legal layer
- the direct text for every lower surface
- the place where repo-native structure originates

### 2. Distill semantically, not literally

The correct move is the same one already required for the imported live-ledger taxonomy:

- absorb the semantics
- preserve the current shell's physical architecture
- do not copy the source surface's packaging literally

As the live-ledger doctrine says of imported directory trees, the shell should translate semantically while keeping physically justified structure. The same rule applies here to instruction surfaces.

### 3. Descend from prose to enforcement

As behavioral law moves downward:

- high-level invariants remain prose
- harness-native execution rules become imperative instructions
- repeated workflows become playbooks and skills
- hard gates become hooks, permissions, validators, and templates

The rule is:

`if a lower surface can enforce it, stop restating it as philosophy and encode it structurally`

### 4. One behavioral core, many projections

The shell should not allow parallel personalities to form in `AGENTS.md`, `CLAUDE.md`, skills, hooks, and templates.

There should be:

- one constitutional behavioral core
- multiple environment-specific projections
- explicit source-vs-generated law
- validator-backed drift detection

### 5. Progressive disclosure is mandatory

CLI environments have better file access, config hierarchy, rules files, skills, hooks, slash commands, and validators than chat apps do.

That means the chat-app CI habit of packing many concerns into one persistent text block should not be copied downward.

In CLI:

- universal law belongs high
- harness doctrine belongs in playbooks
- task-specific procedure belongs in skills or commands
- risky boundaries belong in settings, permissions, and hooks
- output expectations belong in templates

## Distillation Model

### Layer 0: Witness Corpus

Inputs preserved exactly for provenance:

- imported chat-app CI texts
- provenance report
- Sigma doctrine interpreting them

Purpose:

- preserve the original high-resolution captures
- support future parity review when vendor surfaces change
- prevent memory-based drift

### Layer 1: Constitutional Atoms

This is the first required distillation layer and the one that is still too implicit today.

The behavioral kernel should be decomposed into atoms such as:

| Atom family | Example atom | Nature |
| --- | --- | --- |
| Epistemic discipline | separate observation / inference / speculation | cross-surface invariant |
| Response calibration | scale output depth to task complexity | cross-surface invariant |
| Collaboration posture | honest pushback over servility | cross-surface invariant |
| Style | direct, active, affirmative prose | cross-surface invariant |
| Anti-pattern suppression | no decorative verbosity, no hidden certainty, no fake agreement | cross-surface invariant |
| Cognitive accommodation | structure for clarity, chunking, externalization, continuity | cross-surface invariant |
| Repo sovereignty | durable truth in files, not threads | repo-native invariant |
| Progressive disclosure | load only what is needed | CLI-native invariant |
| Trust boundary | deny / allow / ask and policy hooks | harness-native invariant |
| Artifact law | one artifact class, one lane | repo-native invariant |

These atoms should be expressed once at the constitutional or Sigma level, then projected downward.

### Layer 2: Projection Spec

Each atom needs an explicit projection rule:

| Projection question | Required answer |
| --- | --- |
| Which surfaces receive this atom? | `AGENTS`, `CLAUDE`, playbook, skill, hook, template, settings |
| In what form? | prose, checklist, frontmatter, JSON rule, script gate, template field |
| At what compression level? | full, summary, pointer-only, or absent |
| Is it generated or hand-authored? | source or render target |
| What validates parity? | renderer, schema check, lint, path validator, anti-drift test |

Without this layer, the shell will keep oscillating between overstuffed constitutions and under-specified operational surfaces.

### Layer 3: Rendered Surfaces

Rendered or maintained outputs then fall into six main classes.

### `AGENTS.md`

Role:

- sovereign repo constitution
- stable authority spine
- shell-wide invariants
- lane law
- office law
- root law

What belongs here:

- only the stable, cross-surface behavioral core
- source-vs-generated law
- artifact and lane classification
- office and harness differentiation at a constitutional level
- root cleanliness and repo sovereignty

What does not belong here:

- vendor-specific field packing
- long task workflows
- slash-command procedures
- hook internals
- template wording

Compression rule:

- least compressed semantically
- most compressed operationally

In other words: `AGENTS.md` should state the law, not all the ways the law gets executed.

### `CLAUDE.md` and other CLI constitutions

Role:

- thin harness adapter
- imperative contract for a specific tool surface
- entry veneer over deeper doctrine

What survives here:

- the same behavioral kernel
- source-vs-generated reminders
- harness-native loading order
- pointer-rich references to root operator surfaces and relevant playbooks

What compresses differently here:

- chat-app identity scaffolding becomes office posture and modality
- verbosity instructions become response/output discipline
- collaboration posture becomes execution stance
- cognitive accommodation becomes pointer-rich loading and clean structure

What becomes explicit structure around it:

- native config hierarchy
- rules files
- settings precedence
- project/user/local separation

Net rule:

- `CLAUDE.md` should be thinner and sharper than chat CI because CLI already has files, settings, rules, skills, and hooks to carry the rest

### Playbooks

Role:

- operational doctrine
- task and harness patterns
- failure modes
- examples
- load order and escalation rules

What belongs here:

- expanded interpretation of constitutional atoms
- harness grain
- anti-pattern catalogs
- execution heuristics
- context-loading strategy
- examples of good and bad application

Playbooks are where the shell explains:

- why the rule exists
- when to apply it
- how it changes under a given harness

They should not become constitutional clones.

### Skills

Role:

- on-demand bounded protocol
- reusable task module
- progressive-disclosure vehicle

What belongs here:

- narrowly scoped workflow instructions
- preconditions
- inputs and outputs
- required evidence
- file pointers
- verification steps

What gets compressed:

- general behavioral prose collapses into terse operational instructions
- broad cognitive accommodation becomes explicit chunking, checklists, and handoff structure
- anti-pattern reminders become concrete "do/don't" steps

Skills should inherit the constitutional posture implicitly and only restate what is task-critical.

### Hooks and permissions

Role:

- executable boundary
- audit gate
- policy enforcement

What belongs here:

- things that must always fire
- deny/allow/ask policy
- validation of unsafe transitions
- capture of state that would otherwise be lost
- policy logic too strict or too repetitive for prose

Hooks are where these CI-family ideas stop being prose and become machinery:

- "do not bluff completion" -> completion gate requiring evidence
- "separate observation from inference" -> structured log or template validation
- "avoid dangerous actions" -> deny rules and preflight hooks
- "preserve continuity" -> pre-compact and stop hooks that externalize state

Anything that can be checked mechanically should migrate here instead of being re-preached in every instruction surface.

### Templates

Role:

- output mold
- continuity scaffold
- artifact-class normalizer

What belongs here:

- required sections
- metadata fields
- evidence slots
- summary depth expectations
- handoff and receipt structure

Templates are the preferred landing zone when a chat instruction is really an output-shape preference rather than a behavioral law.

Example:

- "be clear, structured, and easy to scan" should not remain a vague stylistic plea if the shell can instead require `Summary`, `Output`, and `Evidence` sections in a result template

## What Survives Unchanged

These principles should survive almost verbatim across all surfaces:

1. Match response depth to actual task complexity.
2. Separate observation, inference, and speculation when epistemic status matters.
3. Use direct, active, affirmative prose.
4. Suppress known anti-patterns explicitly.
5. Prefer honest pushback over flattering compliance.
6. Externalize durable state instead of trusting session memory.
7. Keep truth in repo artifacts rather than platform-local memory.

These are the constitutional atoms of the CI family.

## What Compresses Differently In CLI

CLI surfaces have native affordances that chat-app custom instructions do not.

Because of that, the following transformations should happen:

| Chat-surface burden | CLI translation |
| --- | --- |
| one persistent prose block carries many concerns | split across constitution, playbook, skill, settings, and hook |
| identity/persona paragraph nudges behavior | office posture, modality, and routing law |
| verbosity request | explicit output contract, templates, and result sections |
| careful-chat wording | imperative project instructions and command rules |
| memory-preservation prose | repo-as-memory doctrine, handoffs, ledgers, and hooks |
| safety reminders in prose | permissions, validators, deny rules, and audit hooks |
| user cognitive profile paragraph | chunked templates, pointer-rich context injection, progressive disclosure |

The CLI gain is not just shorter prose.
It is stronger alignment because the behavior moves into native surfaces the tool can actually use.

## What Becomes Explicit Structure Rather Than Prose

The following should become structured surfaces wherever possible:

| Concern | Preferred form |
| --- | --- |
| source vs generated | renderer comments, generation pipeline, validator |
| trust boundaries | `.claude/settings.json` permissions and MCP config |
| always-fire policy | hooks and validators |
| task invocation | slash commands and skills |
| output shape | templates with required fields |
| artifact classification | naming law, directory law, metadata checks |
| parity and drift | config validator and render check |
| provenance | render ledger or config registry |

The operative law is:

`prose should explain the structure, not substitute for missing structure`

## What Must Never Be Copied Literally From Chat Surfaces

The following should not be pasted directly into repo-native CLI artifacts:

1. Vendor-UI packaging logic such as two-field splits, agent/global partitions, or app-vs-studio bifurcation.
2. Consumer-chat compensation wording that exists only because a hosted chat surface tends to under-read or over-perform persona.
3. Long identity scaffolds whose real job is to coax tone rather than govern repo behavior.
4. Redundant anti-pattern lists copied into every artifact family.
5. Cognitive-profile paragraphs when the actual need is better solved by templates, chunking, ledgers, or progressive disclosure.
6. Surface-specific disclaimers that belong in provenance doctrine rather than in everyday operating files.
7. Any prose that a hook, validator, permission rule, or template can enforce more reliably.

This is the core quarantine rule:

`do not copy chat compensation text into CLI law when the CLI has a native mechanism for the same job`

## Net Rule

The essence of the chat-app CI family should survive as constitutional atoms and behavioral invariants.

Its chat-surface wording should not survive unchanged except where the wording itself names a truly cross-surface invariant.

Everything else should be translated downward according to native grain:

- law in `AGENTS.md`
- thin harness contract in `CLAUDE.md`
- doctrine in playbooks
- bounded protocol in skills
- enforcement in hooks and permissions
- output shape in templates
- parity and provenance in validators and ledgers

That is the correct distillation law for turning high-resolution chat custom instructions into a CLI and repo-native configuration architecture.
